## 0. 目的とスコープ

操作PCから任意のHID（マウス/キーボード/タッチ/ゲームパッド … 最大35種）で
被操作デバイス（PC・スマホ・Switch 等）を操作する。ESP32-S3 は**中身を解釈しない
透過ブリッジ**に徹し、「どう動かすか」のロジックは全て操作PC側クライアントに置く。
これにより **ESP32 のファームは原則一度書いたら不変**、機能追加は操作PC側だけで完結する。

- 対象: ESP32-S3（ネイティブUSB = Full-Speed）+ Adafruit TinyUSB
- 既存資産: `hid_descriptor.h`（35 Usage Page を Report ID 1..35 で内包）、`README.md`
- 本書の範囲: ①UART通信プロトコル ②ESP32ファーム設計 ③操作PCクライアント設計
  ④レイアウトの単一情報源 ⑤UX ⑥横断的関心事 ⑦マイルストーン ⑧未決定事項

## 1. システム全体像

```
 ┌───────────────┐   UART (framed, 1Mbps)   ┌────────────────┐   USB HID (native FS)  ┌────────────────┐
 │ 操作PC          │  ──────── cmd ─────────▶ │  ESP32-S3        │  ───── HID report ───▶ │ 被操作デバイス     │
 │ + Client Lib    │ ◀─── status / event ──── │  透過ブリッジ      │ ◀──── out/feature ───  │ PC/スマホ/Switch │
 │ (Python等=脳)    │                          │ (反射神経+安全装置) │  列挙時に desc_hid_report[] を提示 │ (USBホスト)      │
 └───────────────┘                          └────────────────┘                        └────────────────┘
```

| 役割 | 構成要素 | 責務 |
|---|---|---|
| **脳** | 操作PC + クライアントライブラリ | 何を・いつ送るかの全ロジック。HIDレポートの組み立て。状態管理。 |
| **反射神経** | ESP32-S3 ファーム | UART受信→検証→`sendReport` 中継のみ。安全装置（後述）。中身は非解釈。 |
| **顔** | `hid_descriptor.h` | 被操作デバイスに「自分は何者か」を申告（列挙時）。Report ID/レイアウトの契約。 |

**重要原則**：被操作デバイスが会話する相手は ESP32 だけ（操作PCは不可視）。
ゆえに「正体（記述子）」は ESP32 側に常駐必須、「中身（ペイロード）」は操作PCが決める。

## 2. 用語

- **Target / 被操作**: USBホスト側。ESP32 を HID として認識する。
- **Operator / 操作PC**: UART で ESP32 に指令を送る側。
- **Report**: HID の送信単位。`[Report ID][フィールド値...]`。最小アドレス単位は Report ID。
- **Full-state（全状態）送信**: 差分でなく、そのレポートの現在値を毎回まるごと送る方式。

## 3. 通信プロトコル（Operator ⇄ ESP32, UART）

### 3.1 設計方針
- **COBS フレーミング**：0x00 を区切りに使う無曖昧方式。ペイロードに任意バイト（0xAA等）が
  出ても誤同期しない。欠落しても次の 0x00 で確実に再同期。
- **CRC-16**：加算和より検出力が高い。化けたフレームは破棄。
- **双方向**：コマンド（操作→ESP32）と、状態/イベント（ESP32→操作）。
- **接続順非依存**：READY を一方通知に頼らず、PING で**問い合わせ可能**にする。

### 3.2 フレーム形式（COBSエンコード前の論理フレーム）

```
+--------+--------+----------------------+----------+
| TYPE   | SEQ    | PAYLOAD (0..N)       | CRC16    |
| 1 byte | 1 byte |                      | 2 bytes  |
+--------+--------+----------------------+----------+
  └─ CRC16 は TYPE..PAYLOAD を対象。全体を COBS 化し、末尾に 0x00 区切りを付けて送出。
```

- SEQ: 0..255 ローテーション。欠落検知・任意のACK照合に使用。
- SEND_REPORT の最大ペイロード = ReportID(1) + HIDデータ(最大63) = 64B（§8.1 の63B上限に対応）。

### 3.3 メッセージ種別

**操作PC → ESP32**

| TYPE | 名称 | PAYLOAD | 用途 |
|---|---|---|---|
| 0x01 | SEND_REPORT | `[ReportID][data...]` | HIDレポートを1つ注入 |
| 0x02 | PING | — | 生存/準備状態の問い合わせ |
| 0x03 | RELEASE_ALL | — | 全レポートをゼロ送信（パニック/安全解除） |
| 0x04 | GET_CAPS | — | 利用可能な Report ID とサイズ一覧を要求 |
| 0x05 | SET_IDENTITY | `[VID:2][PID:2][bcd:2][profile:1]` | VID/PID/プロファイル変更（再列挙する） |
| 0x06 | SET_FEATURE_CACHE | `[ReportID][data...]` | Feature 即答用キャッシュを供給（双方向） |

**ESP32 → 操作PC**

| TYPE | 名称 | PAYLOAD | 用途 |
|---|---|---|---|
| 0x81 | STATUS | `[flags][...]` | mounted/ready/suspended。変化時とPING応答で送出 |
| 0x82 | ACK | `[seq]` | SEND_REPORT受理（任意の信頼配送モード時） |
| 0x83 | NACK | `[seq][reason]` | 拒否（CRC不正/長さ不正/未知ID/未mount） |
| 0x84 | HOST_EVENT | `[ReportID][data...]` | 被操作→デバイスの Output/Feature を逆中継（例: LED状態） |
| 0x85 | LOG | `[text]` | デバッグ/エラー |

STATUS flags（例）: bit0=mounted, bit1=suspended, bit2=ep_ready, bit3=watchdog_fired。

### 3.4 信頼性モデル（2モード）
- **既定: 投げっぱなし（fire-and-forget）+ 全状態送信**。単発の欠落はクライアントが
  毎回全状態を送ることで**次フレームで自己回復**（§8.2）。実装が軽く実用十分。
- **任意: 信頼配送**。クライアントが SEQ ごとに ACK 待ち、タイムアウトで再送。
  確実性が要る局面（重要な単発操作）で有効化。

### 3.5 安全装置（プロトコルに内蔵）
- **ウォッチドッグ自動リリース**：直近の有効フレームから T ms 無通信、かつ
  非ゼロ（押下中）のレポートがあれば、ESP32 が自動的に**当該レポートをゼロ送信**して
  キー/ボタン押しっぱなしを解除。操作PCのクラッシュやケーブル抜けでの暴走を防ぐ。
- **unmount 時リリース**：被操作が外れたら状態を破棄し ready=false。

## 4. ESP32-S3 ファームウェア設計（モジュール構成 / 状態機械）

```
            ┌──────────────────────────── main loop ───────────────────────────┐
 UART RX ─▶ │ [COBS/CRC 復号] ─▶ [Dispatcher] ─▶ handlers                        │ ─▶ usb_hid.sendReport()
            │                         │  ├ SEND_REPORT: 検証→ready判定→送信→状態記録 │
            │                         │  ├ PING:        STATUS 応答                  │
            │                         │  ├ RELEASE_ALL: 全live状態をゼロ送信           │
            │                         │  └ GET_CAPS:    ReportID/サイズ表を返す        │
 UART TX ◀─ │ [COBS/CRC 符号] ◀─ [STATUS/ACK/EVENT/LOG]                          │
            │ [Watchdog]  [Mount監視] [Auto-release]                              │
            └────────────────────────────────────────────────────────────────────┘
```

| モジュール | 役割 |
|---|---|
| USB Init | `usb_hid.setReportDescriptor(desc_hid_report, sizeof)` → `begin()`。任意でVID/PID設定。 |
| UART Link | `Serial0`（UART0, 既定GPIO43/44）@ 1Mbps。RXバッファ拡張。※native USB は TinyUSB 専有 |
| Framing | COBS復号 + CRC16検証 / 送信は符号化 |
| Dispatcher | TYPE で分岐 |
| SEND_REPORT Handler | ①ReportIDが既知 ②payload長 ≤ 63 かつ宣言サイズと一致 ③`usb_hid.ready()` を満たす時のみ送信。送信値を `last_state[ReportID]` に記録 |
| Safety/Watchdog | 無通信タイムアウト & unmount 時に押下中レポートを自動リリース |
| Handshake | mount変化で STATUS 自動送出 + PING応答（順序非依存） |
| Caps | `ReportID → {size,type,usage}` 表を返す（§6で単一情報源化） |

ファーム状態機械（概念）: `BOOT → WAIT_MOUNT → READY ⇄ SUSPENDED`、各遷移で STATUS 送出。

## 5. 責務境界（ESP32ファーム vs クライアント）

**原則：ESP32＝機構(mechanism)、クライアント＝方針(policy)。** ESP32 は“鈍く安定”、頭脳は全部クライアント。

| 領域 | ESP32ファーム（機構） | クライアント（頭脳） |
|---|---|---|
| USB列挙・記述子提示 | ● `desc_hid_report[]` を target に提示、mount管理 | ― |
| レポートのバイト構築 | ―（非解釈で中継のみ） | ● `reports.json` 準拠で組立 |
| 論理状態・全状態送信 | ―（最終値は safety 用に記録のみ） | ● 状態保持・全状態送信 |
| UART転送（COBS/CRC/ACK） | ● フレーミング・整合性・再送機構 | ● 同プロトコルの相手側 |
| 検証（ID既知/長さ≤63/ready） | ● 門番として拒否（NACK） | ● そもそも不正を送らない |
| 安全（watchdog自動リリース等） | ● **操作PC不在でも作動** | ○ RELEASE_ALL を要求可 |
| 双方向: GET_REPORT(Feature)即答 | ● キャッシュから即答 | ● キャッシュ内容を供給＝**真実源** |
| 双方向: Output(LED等)受信 | ● HOST_EVENT で上げる | ● 受けて解釈・反応 |
| identity / プロファイル | ● 記述子群保持・**再列挙の実行** | ● どれを使うか選択・指示 |
| 自動化/マクロ/タイミング/文字列入力 | ― | ● 全ロジック |

**迷ったときの物差し**：①操作PC不在でもリアルタイムに要る→ファーム（安全） ②USB/ターゲット状態を知る必要→ファームが報告/クライアントが判断 ③何を/なぜ/いつ→クライアント ④バイトを壊さず届ける→ファーム（転送）。

**双方向の肝**：host の `GET_REPORT(Feature)` は即答が要る（UART往復は遅すぎ/レース）。よって **ファームが Feature 値のキャッシュを保持し、クライアントが `SET_FEATURE_CACHE` で事前供給**。Output(LED等)は非同期なので `HOST_EVENT` で上げてクライアントが反応。→ Feature の“内容”はクライアントが真実源、ファームは即答用キャッシュ＋転送に徹する。

### 5.x クライアント高レベルAPI（保留・統一方針）

- クライアントの高レベルAPIは**本フェーズでは設計しない**（プロトコル確定後に後付け可能）。
- 設計時も**キーボード/マウス等を特別扱いしない**。全35種を**同一の汎用I/F**で扱う：
  - コア＝`send_report(report_id, bytes)` / `on_event(...)` の汎用パイプ。
  - 型が要るなら `reports.json` 駆動の**汎用 Report オブジェクト1個**（`r=report(ID); r.set("field",v); r.send()`）を全種共通で。`Keyboard`/`Mouse` クラスは作らない。
  - `type("...")` 等は任意の外付けヘルパー（コアは関知しない）。

## 6. レポートレイアウトの「単一情報源」（重要）

操作PCと ESP32 が**同じバイト配置**を共有しないと 1:1 が崩れる。`hid_descriptor.h` を唯一の真実とし、
そこから機械生成する：

```
 hid_descriptor.h ──[ビルド時パーサ]──▶ reports.json (ReportID → {size, fields[], usage})
                                          │
                          ┌───────────────┼────────────────┐
                          ▼                                ▼
              firmware: ReportID→size 表        client: 各デバイスのエンコーダ/デコーダ
```

- 二重定義を避け、descriptor を直せば両側が自動追従。
- 当面は手書き表でも可（小さい）。本格運用ではコード生成を推奨。

## 7. UX とシーケンス

**ユーザー体験**：①操作PCと被操作デバイスを ESP32-S3 に挿す → ②クライアントを起動 →
③`bridge.keyboard.type(...)` 等で被操作を操作。

```
操作PC            ESP32                  被操作
  │                 │   ◀── enumerate ──── │   (被操作を接続 / 既に接続済み)
  │── PING ───────▶ │                      │
  │◀ STATUS(ready) ─│                      │
  │── SEND(kbd,'A')▶│──── HID report ────▶ │   A 入力
  │── SEND(kbd, 0)─▶│──── HID report ────▶ │   離す（全状態送信）
  │   (操作PCクラッシュ / ケーブル抜け)         │
  │      watchdog → 当該レポートをゼロ送信 ───▶ │   自動リリース（押しっぱなし防止）
```

接続順は**どちらでも可**（ハンドシェイクが吸収）。被操作→操作の順なら、操作PC側は接続後に
PING で READY を確認して開始。

## 8. 横断的関心事（これまでの教訓の反映）

### 8.1 1レポート63バイト上限
- `CONFIG_TINYUSB_HID_BUFSIZE=64`（ID込み）→ 実データ最大63B。kbd/mouse/gamepad等は余裕。
- 超過する3種（Monitor EDID 130B / MSR 229B / FIDO 65B）はバッファ拡張（ESP-IDF）か対象外。
- クライアント・ファーム双方で**長さ検証**（63B超は送らない）。

### 8.2 HIDは状態を持つ → 全状態送信＋ウォッチドッグ
- 欠落で「離す」が消えると押しっぱなし事故。**毎回全状態を送る**ことで自己回復。
- さらにファームの**自動リリース**で二重に保護。

### 8.3 ホストの選り好み → 記述子プロファイル
- 35-in-1 は「何でも送れる」反面、Switch 等の厳格ホストに弾かれる可能性（実測前提）。
- 対策: **プロファイル**＝用途別の記述子（universal / switch-gamepad / keyboard-only …）を用意。
  当面はビルド時切替、将来は SET_IDENTITY+再列挙で実行時切替。

### 8.4 デバイス同一性（VID/PID/文字列）
- コンソール認識や特定機器偽装には VID/PID が要る。SET_IDENTITY 経路を将来用に確保（再列挙必須）。

### 8.5 給電・接続順
- 多くのデュアルポート開発ボードはダイオードORで両ポート給電（**基板依存・要確認**）。
- 「未mountでフリーズ」は誤り（`ready()`がfalseなら送らないだけ）。ハンドシェイクは
  「取りこぼし防止」と「操作PCへの準備通知」が目的。

### 8.6 FIDO は対象外（本ブリッジでは）
- CTAPHID は Report ID 無し・厳密64Bが必須。共有インターフェースでは不可。
  必要時は専用HIDインターフェースを別立て（`CFG_TUD_HID=2` 確保済み）。

### 8.7 安全な利用
- 本質的に強力なHID注入器。用途は自動化・テスト・アクセシビリティ・遠隔操作等の正当目的を想定。
- 安全装置（自動リリース/RELEASE_ALL）は誤操作・暴走対策として常設。

## 10. 確定した方針（全部入り）

本プロジェクトは「全35ページを実用」を目標とし、以下を**すべて採用**する。

| 項目 | 決定 |
|---|---|
| 信頼性 | **ACK/再送あり**（投げっぱなしモードも併設し用途で選択） |
| 双方向 | **あり**（Feature 即答キャッシュ + Output を HOST_EVENT で逆中継） |
| VID/PID・プロファイル | **あり**（SET_IDENTITY + 記述子プロファイル + 再列挙） |
| クライアント言語 | **Python**（pyserial、まず実装）。プロトコルは言語非依存 |
| クライアント高レベルAPI | **保留・統一I/F**（§5。特別扱いしない） |
| レイアウト単一情報源 | **コード生成**（`hid_descriptor.h` → `reports.json` / `mh_reports.h`） |
